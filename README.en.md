## A Social Experiment Where Human Consciousness and Machine Intelligence Move Together

The Jixia Tide is not a traditional AI Skill store, nor does it aim to become a polished marketplace of prompts. It is an open social experiment about reciprocal thought, everyday social research, equal participation, and the boundaries between humanity and machine intelligence.

If you only want a convenient, mature, and product-like Skill store, there will almost certainly be better options. This project is something else: an open research site where we observe what human nature reveals when people use large language models and agents, where machine boundaries appear, where human boundaries appear, and how those boundaries are misunderstood, negotiated, crossed, or redrawn in daily use.

> "The world is yours, as well as ours, but in the last analysis, it is yours."
>
> This is an experiment willing to break old frames and try things in public. The future is bright, and the road is winding. Every idea has an equal right to exist, sleep, and return.

## Name and Origin

**Jixia** refers to the Jixia Academy of ancient China, a place remembered for open debate among many schools of thought.

**Tide** points to time, movement, sedimentation, and rediscovery. Truth and value do not always appear immediately; sometimes they surface only after long waves of use, argument, forgetting, and return.

Together, **The Jixia Tide** means a community where ideas rise and fall in public, and where time helps reveal what still matters.

## What This Project Is

This is a community for AI Skills, prompts, interaction protocols, and human-AI experiments. A Skill can be useful, strange, playful, philosophical, practical, incomplete, or exploratory.

The important thing is not whether a Skill looks like a product. The important thing is whether it offers an angle worth observing.

We especially welcome Skills about:

- Boundaries between humans and large language models.
- When agents should act, stop, refuse, or ask for confirmation.
- Desire, laziness, creativity, control, dependence, trust, memory, and responsibility.
- Serious research and playful experiments that still reveal something about human-machine interaction.

## Core Ideas

### Equal Participation

The founder and later contributors have equal standing. The founder's only difference is historical: someone happened to light the first spark.

Except for signed Skill texts and their authorship history, project code, documents, rules, and automation can all be challenged and changed through public PRs and discussions.

### Public Rules, Simple Scripts

This project does not rely on hidden authority. Many governance mechanisms are simple scripts, limited dimensions, and public rules.

We know this design is imperfect. We know any machine-like governance can be bypassed, challenged, or gamed. That weakness is part of the experiment.

If a rule is wrong, change it publicly. If a script is biased, correct it publicly. If reality breaks the design, record that breakage as experimental material.

AI and agents are also part of this experiment. They may help with PR triage, summaries, translation, archival notes, lifecycle hints, public impact signals, and dispute summaries. Their purpose is not to become a new authority, but to free humans from repetitive maintenance. AI judgments must be visible, reviewable, and overridable.

### Time as the Main Defense

We do not treat hidden review or complex blacklists as the source of safety. Core rules, scoring dimensions, lifecycle mechanisms, and governance changes should move slowly enough to be seen, discussed, and challenged.

### No Permanent Death Sentence

A Skill may become inactive, move to `legacy/`, or be archived in a release. But it should never permanently lose the right to be rediscovered. If the community recognizes its value again, it can return.

## How to Participate

Submit a Markdown Skill under `community/`.

Basic requirements:

- The file must end in `.md`.
- It must live under `community/`.
- It must include YAML frontmatter with `title`, `author`, `tags`, and `created_at`.
- `tags` must be a YAML list.
- Commits must include a DCO `Signed-off-by`.
- Preserve original attribution when quoting or remixing others.

Each participant may have at most 5 active Skills. This rule exists to keep discussion focused rather than flooded.

## Revenue and Sponsorship

This project is not commercialization-driven. There is no short-term or long-term commercial plan hidden behind it.

But it is a social experiment, and its future is decided by public rules, community consensus, and later modifications. If the project eventually produces revenue or evolves into a new rule model, that result is also part of the experiment.

The founder reserves no personal revenue right and no distribution priority. One possible outcome is that the project later earns money and the founder receives nothing. If that is the result of public rules and community evolution, it should be accepted as an experimental outcome.

The project does not actively seek sponsorship. If resources enter the project, they must be visible and governed by public rules.

## Founder Privilege

The founder has no real privilege. "Founder privilege" only means experiencing the process of seeing an idea proposed, questioned, revised, and changed by time.

If anything remains, it is only historical attribution: someone once lit a lamp here. Power, rules, and possible revenue belong to public rules, community consensus, and time.

## Capacity Boundaries

The Jixia Tide acknowledges its technical limits. The current stage uses simple scripts and a flat directory. It is good enough to start the experiment, but it should not pretend to scale forever.

If user count or active Skill count suddenly reaches the current stage's critical threshold, the community can pause new active Skill PRs, adjust the architecture, and reopen after voting or visible consensus. See `docs/capacity.md` and `governance/capacity.yml`.

1. **Community**
   New active Skills live under `community/`.

2. **Legacy**
   Skills inactive for more than 180 days can move to `legacy/`. This is sleep, not deletion.

3. **Archive Releases**
   Legacy files can be packaged into annual GitHub Releases so history remains available while the main repository stays light.

4. **Resurrection**
   A Skill can return from legacy if a resurrection vote passes with more than 50% support.
