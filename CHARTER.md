# 稷下观澜自治宪章

语言 / Languages: [简体中文](CHARTER.md) | [English](CHARTER.en.md) | [多语言说明](docs/i18n.md)

原文为中文。若其他语言翻译导致语义理解有误，欢迎通过 Issue 或 PR 共同维护修正。

## 0. 宪章定位

稷下观澜不是一个旧式 Skill 商店，也不是一个追求标准化交付体验的产品目录。它是一场关于思想互惠、日常社会性研究、人性与机器性边界的开源实验。

本宪章用于说明社区如何运行。README 负责表达精神，自动化脚本负责执行部分规则，本宪章位于两者之间：它把价值观翻译成可讨论、可修改、可执行的明文规则。

## 1. 根本原则

1. **平权共生**
   发起者、贡献者、后来者在社区治理中没有天然等级。发起者只代表历史起点，不代表永久权力。

2. **代码即法律，规则可改**
   社区优先依靠公开脚本、公开规则和 GitHub 原生记录运行。任何规则都可以被提出修改；规则修改本身也必须公开留下痕迹。

3. **阳谋，不是阴谋**
   本项目的治理逻辑并不神秘，很多时候只是简单脚本、有限维度和公开判断。我们承认它会有缺陷，也把被现实击穿、修正和再实验的过程视为项目的一部分。

4. **时间见证**
   不用隐蔽审查制造安全感。核心规则、判断维度、生命周期机制的改变，应经过充分讨论，而不是在短期争议中被仓促改写。

5. **永不永久判死刑**
   Skill 可以沉底、休眠、归档或暂时失去活跃位置，但不应被永久剥夺复活权。只要社区重新认可，它就应有回到活跃区的路径。

6. **稳定优先但不牺牲平权**
   安全修补、法律风险处理和攻击防护的目标是维持服务连续性与用户体验，不得把项目改造成隐藏特权或少数人任意裁决的系统。

## 2. 参与者与发起者

### 2.1 参与者

任何人都可以提交、讨论、质疑、衍生或改进项目规则。参与者不因先来后到、身份、名气、技能水平而获得天然优先级。

### 2.2 发起者

发起者没有真正的特权。所谓“发起者特权”，只是享受这个观念被提出、被质疑、被修正、被时间冲刷的过程。

若一定要留下什么，也只是一个历史署名：这里曾经有人点过一盏灯。除此之外，权力、规则和可能出现的收益，都交给公开规则、社区共识与时间。

## 3. Skill 提交规则

### 3.1 文件规则

Skill 必须满足以下基础条件：

- 文件必须是 `.md` 格式。
- 文件必须位于 `community/` 目录。
- 文件必须包含 YAML Frontmatter。
- Frontmatter 至少包含 `title`、`author`、`tags`、`created_at`。
- `tags` 必须是 YAML 列表。
- 提交 commit 必须包含 DCO `Signed-off-by`，确认贡献者有权提交并同意项目许可证。

### 3.1.1 许可证

本项目采用双许可证结构：

- 代码、脚本、工作流配置使用 MIT License。
- 文档、宪章、项目说明和 Skill 文本使用 CC BY-SA 4.0。

贡献者提交内容即表示同意这些许可证。具体说明见 `LICENSE`、`LICENSE-DOCS.md`、`CONTRIBUTING.md` 与 `DCO.md`。

### 3.2 内容方向

Skill 不必像商品、模板或标准答案。它可以是工具、角色、流程、思想框架，也可以只是一个观察角度、一段使用经验、一种人机关系的试探。

社区尤其鼓励以下方向：

- 人与 AI 大模型之间的理解边界。
- Agent 何时应该行动、停下、拒绝或请求确认。
- 人性、机器性、欲望、懒惰、控制欲、依赖感、信任感的日常观察。
- 严肃研究与轻松试验并存的“为了玩”的探索。

### 3.3 数量限制

每位参与者最多提交 5 个活跃 Skill。这个限制的目的不是压制表达，而是避免数量淹没讨论，鼓励每个 Skill 都有清晰角度。

Gatekeeper 会基于 `author` frontmatter 统计 `community/` 中的活跃 Skill 数量，并拒绝会让同一作者超过 5 个活跃 Skill 的 PR。

## 4. Skill 所有权与衍生

### 4.1 原作者署名

原作者署名的 Skill 文本承载个人表达与历史，默认不由他人直接改写。任何引用、衍生、改造都应保留原作者署名。

### 4.2 如何演化他人 Skill

如果你想修改他人的 Skill，优先选择以下路径：

- 提出 Issue，说明建议与理由。
- 创建衍生 Skill，保留原作者来源。
- 等待原作者主动修订。

严禁恶意覆盖、删除或冒名改写他人的 Skill。

### 4.3 项目公共部分

除个人署名 Skill 文本及其历史外，项目代码、文档、规则、自动化逻辑都可以被社区通过 PR 讨论、修订、重写。

## 5. 自动化治理

### 5.1 Gatekeeper

Gatekeeper 负责 PR 的基础门禁：

- 只接受 `community/` 下的 Markdown Skill。
- 校验必需 frontmatter。
- 扫描明显恶意内容。
- 校验通过后自动批准并尝试合并。
- 校验失败时必须公开评论原因。

Gatekeeper 只能执行主分支受信任脚本，不得执行贡献者分支中的代码。

### 5.2 Time Sentinel

Time Sentinel 负责生命周期沉底：

- 每天定时扫描 `community/`。
- 参考 Git 最后提交时间、`updated_at`、`last_used_at`、`last_interaction_at`。
- 超过 180 天无更新、互动或使用记录的 Skill，可移入 `legacy/`。
- 沉底不是删除，也不是惩罚，而是进入休眠期。

### 5.3 Chronicler

Chronicler 负责编年史归档：

- 每年归档 `legacy/` 中的文件。
- 生成 `Archive_{Year}.zip`。
- 发布 GitHub Release，tag 为 `archive-{year}`。
- 发布成功后清理 `legacy/`，保持主仓库轻量。

历史不应消失。即使从主仓库清理，也应能在 Release 中被重新发现。

### 5.4 Resurrection Council

Resurrection Council 负责复活机制：

- 任何人可以通过 Issue 评论 `/vote-resurrect {skill_name}` 发起复活投票。
- 投票持续 7 天。
- 赞成票比例超过 50% 时，Skill 可从 `legacy/` 回到 `community/`。
- 复活结果必须在原 Issue 中公开回复。

### 5.5 AI 与 Agent 维护

本项目鼓励使用 AI 与 Agent 承担维护劳动。AI 可以做机械检查、摘要、翻译、重复内容提示、PR 初审、生命周期建议、影响力信号整理、归档说明和争议整理。

AI 与 Agent 的目标是把人从重复维护中解放出来，让人主要关心自己的 Skill 是否有价值、是否被阅读、是否需要迭代，以及如何参与社区活动。

AI 的判断必须可见、可复查、可推翻。AI 不能拥有隐藏特权，不能绕过宪章封禁参与者，不能私下改变规则，不能成为新的权威。

如果 Agent 的实现逻辑被社区修改，导致系统短暂瘫痪、误判或行为变化，这也是实验的一部分。只要没有触及法律、平台规则、安全和稳定性底线，应优先通过公开 PR、Issue 和回滚机制修复，而不是引入隐藏的人治特权。

AI 维护失败本身也应被记录为实验数据。

### 5.6 Influence Sentinel

Influence Sentinel 负责生成公开影响力信号：

- 统计来源只能是 Git 历史、`community/*.md`、Issue、PR、评论、Reaction 和复活投票等公开可审计对象。
- 输出 `community/index.json` 作为机器可读索引，输出 `docs/impact.md` 作为人类可读实验面板。
- 分数只能作为观察、排序和维护建议，不得直接导致封禁、剥夺复活权或改变作者平等身份。
- 评分公式必须公开。若某项信号被滥用，应公开调整公式或权重，而不是引入隐藏算法。

## 6. 收益与赞助

本项目不以商业化为目的，没有短期与长期之分。

但项目也是一场社会性实验。系统最后会变成什么样，由公开规则、社区共识和后续修改共同决定；规则本身也可以被更改。如果未来项目真的产生现实收益，甚至发展出某种新的规则模式，那也是实验的一部分。

发起者不预留个人收益权，也不预设分配优先级。项目存在一种完全可能的结局：它后来盈利了，但发起者一分钱也没有。只要那是公开规则和社区演化的结果，就应被视为实验结果。

项目暂不主动接触赞助，也不让赞助成为规则制定的隐藏力量。真有资源进入时，必须摆在明面上，由公开规则约束。

## 7. 规则修改

### 7.1 可修改范围

本宪章、README、自动化脚本、目录结构、生命周期规则和收益规则都可以被修改。

### 7.2 修改方式

规则修改应通过公开 PR 或 Issue 讨论进行。涉及以下内容的修改，应尽量给出更充分的讨论时间：

- Skill 所有权。
- 收益与赞助。
- 复活门槛。
- 沉底周期。
- Gatekeeper 自动合并范围。
- 发起者与参与者权利。

### 7.3 慢速决策

核心规则不应在短期情绪中被仓促改写。若规则争议较大，默认放慢，而不是强行加速。

## 7.4 容量边界与暂停机制

本项目承认当前技术路径存在容量上限。当前阶段使用平铺 `community/` 目录和简单脚本扫描，适合启动实验，但不假装能够无限扩张。

当用户量或活跃 Skill 数量突然达到当前设计阶段的临界值时，社区可以将 `governance/capacity.yml` 中的 `status` 改为 `paused`。进入暂停状态后，Gatekeeper 应拒绝新的或更新的活跃 Skill PR，但治理、文档、翻译和架构调整 PR 仍可继续。

暂停不是惩罚，而是边界控制。它表示当前架构已经不足以继续安全扩张，需要先讨论下一阶段设计。

架构调整继续由社区维护。任何人都可以提出 registry、作者分片、多仓库、搜索索引或其他方案。只要相关方案经社区投票或公开共识通过，就可以恢复 Skill PR。

如果大家都卡住了，那也是实验的一部分。贡献可能有，也可能没有；很多人本来就是为了玩、为了留下名字、为了观察这个系统会变成什么样。

容量阶段、暂停阈值和恢复条件见 `docs/capacity.md` 与 `governance/capacity.yml`。

## 8. 争议与降级

### 8.1 系统不完美

我们接受系统存在漏洞、误判和局限。简单脚本控制不了复杂社会，有限维度也无法穷尽所有人机互动。

### 8.2 公开修正

当脚本、规则或治理逻辑出现错误时，应优先公开记录问题、讨论修正，而不是隐藏失败。

### 8.3 在实验中学习

本项目允许边做边改。被现实击穿的地方，不只是故障，也是实验材料。

### 8.4 被骂与停服

道德批评、一般争议、误解、嘲讽和围观本身不是停服理由。只要服务没有被法律、平台规则、恶意攻击或稳定性风险击穿，被骂也可以作为社会实验现象被观察。

但法律风险、版权侵权、隐私泄露、骚扰仇恨、恶意软件、凭据窃取、破坏性攻击、平台规则违规和 GitHub Actions 权限滥用必须优先处理。

具体风险模型见 `docs/risks.md`。

## 9. 宪章与脚本冲突

当宪章与当前脚本实现不一致时：

1. 当前脚本代表已经自动执行的现实能力。
2. 本宪章代表社区希望脚本逐步对齐的明文规则。
3. 若冲突影响实际治理，应优先开 Issue 或 PR 修正脚本或修订宪章。

规则不是石碑，而是潮汐中的航标。
