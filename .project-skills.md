# 稷下观澜 Project Skills

本文件是本项目的专属 Skill 宪法。后续开发必须先读取本文件；当本文件规则与通用最佳实践冲突时，优先遵守本文件，除非某条 Skill 已被标记为 `[状态: 异常/待修复]` 或 `[状态: 已废弃]`。

## 初始悟道：隐形架构原则

### Project-Specific-Meta-OriginSecureConstitution

- 状态: 生效
- 适用范围: 本项目所有后续开发对话、代码生成、测试补充、重构与规则演进。
- 核心原则: 用户提供的 base secure 是本项目思想起源；每次开发必须先读取 `.project-skills.md`，将其中 Project-Specific Skill 作为最高开发宪法，并在生成代码前优先推演一个稳健方案。
- 输入参数:
  - `.project-skills.md` 当前内容。
  - 用户本轮请求。
  - 当前仓库代码与 Git 状态。
- 预期输出:
  - 若 `.project-skills.md` 存在: 先遵循其生效规则，再执行开发。
  - 若 `.project-skills.md` 不存在: 初始化该文件，扫描核心代码，提取 1-3 条隐形架构原则。
  - 若出现复杂且预计复用 2 次以上的新项目逻辑: 追加 `Project-Specific-[模块名]-[功能名]`。
- 异常降级:
  - 如果某条 Project-Specific Skill 导致逻辑错误，立即将其在当前会话降级为最低优先级，并在本文件标记 `[状态: 异常/待修复]` 与原因。
  - 如果新灵感高风险但可能有突破性，只能作为备选思路自然提出，不得覆盖稳健方案。

### Project-Specific-Governance-RepositoryAsStateMachine

- 状态: 生效
- 适用范围: 所有 Skill 生命周期治理脚本与 GitHub Actions 工作流。
- 核心原则: `community/`、`legacy/`、GitHub Issue/PR/Release、Bot commit 是系统状态本身；不要引入外部数据库、外部服务器或不可审计的隐藏状态。
- 输入参数:
  - 仓库工作树中的目录状态。
  - GitHub 原生对象状态，包括 PR、Issue comment、Reaction、Release。
- 预期输出:
  - 可由 Git 历史或 GitHub 原生对象审计的状态迁移。
  - 规范 Bot commit，例如 `[Bot] Archive inactive skill: {Skill_Name} to Legacy`。
- 异常降级:
  - 如果 GitHub API 状态不可读取，则脚本必须失败并输出清晰日志，不得猜测状态。
  - 如果目标文件已存在或状态冲突，则拒绝覆盖，保留现状并记录原因。

### Project-Specific-Gatekeeper-UntrustedPullRequestIsolation

- 状态: 生效
- 适用范围: `pull_request_target`、PR 自动审核、自动合并、贡献文件校验。
- 核心原则: Gatekeeper 只能执行主分支受信任脚本，并通过 GitHub API 读取 PR head 文件内容；不得 checkout 后执行贡献者分支中的脚本、配置或安装逻辑。
- 输入参数:
  - PR number。
  - PR changed files 列表。
  - PR head SHA 上的 Markdown 文件内容。
- 预期输出:
  - 校验通过: 添加通过标签、批准 PR、尝试 squash merge。
  - 校验失败: 添加失败标签，在 PR 中逐条评论失败原因。
- 异常降级:
  - 无法读取 PR 文件、mergeable 状态未知超时、自动合并失败时，必须评论说明具体原因并让 workflow 失败。
  - 校验范围必须限制在 `community/*.md`，禁止通过 Gatekeeper 删除文件或修改其他目录。

### Project-Specific-Lifecycle-IdempotentBotActions

- 状态: 生效
- 适用范围: Time Sentinel、Chronicler、Resurrection Council 及后续所有定时/事件型 Bot。
- 核心原则: Bot 行为必须尽量幂等；无事发生时应 no-op 成功，有状态迁移时一次只提交可解释的最小变更。
- 输入参数:
  - 当前 UTC 时间。
  - 文件 frontmatter 中的 `updated_at`、`last_used_at`、`last_interaction_at`。
  - Git last commit date。
  - Issue comment 中嵌入的 HTML marker。
- 预期输出:
  - 无待处理事项: 输出日志并正常退出。
  - 有待处理事项: 完成目录迁移、Release 发布或投票结果处理，并生成规范 Bot commit/Issue comment。
- 异常降级:
  - 单个文件、单个 Issue 或单个 vote 出错时，记录异常并继续处理其他对象。
  - 任何会覆盖现有 Skill 文件的操作必须中止，不得静默覆盖。

### Project-Specific-Docs-MultilingualPublicCanon

- 状态: 生效
- 适用范围: README、CHARTER、CONTRIBUTING、SETUP、许可证说明、公共治理文本及后续多语言文档。
- 核心原则: 本项目面向全球参与者，但中文叙事是当前源文本；多语言版本必须忠实保留“社会实验、平权、规则可改、发起者无特权、不以商业化为目的”的核心含义，不得为了本地化而改写治理承诺。
- 输入参数:
  - 中文源文档。
  - 已存在的目标语言文档。
  - 公共入口文件中的语言索引。
- 预期输出:
  - `README.md` 与 `CHARTER.md` 作为中文源文档。
  - `README.en.md` 与 `CHARTER.en.md` 作为英文全球入口。
  - `docs/i18n.md` 记录翻译规范、语言状态和贡献方式。
  - 公共文档顶部提供明确语言跳转。
- 异常降级:
  - 如果不同语言版本含义冲突，以中文源文档和 CHARTER 的当前版本为准，并开 Issue 修正翻译。
  - 如果某语言无法完整维护，只能标记为草稿或社区维护版本，不得伪装成等价正式文本。

### Project-Specific-Security-ContinuityWithoutHierarchy

- 状态: 生效
- 适用范围: 攻击防护、内容治理、法律风险处理、争议处理、Gatekeeper 策略、社区稳定性设计。
- 核心原则: 安全修补的目标是维护服务连续性和用户体验，不得把项目改成少数人任意裁决的层级系统；道德批评和一般争议可作为实验现象观察，法律风险、平台规则风险、恶意攻击和服务中断风险必须优先处理。
- 输入参数:
  - GitHub 平台规则与法律底线。
  - 当前宪章、README、贡献规则。
  - 自动化脚本的实际执行能力。
  - 攻击、滥用、争议和服务中断信号。
- 预期输出:
  - 明确区分“可被骂但不停服”的争议和“必须处理”的法律/平台/恶意攻击风险。
  - 防护措施优先使用公开规则、自动化、日志、暂停阈值和社区投票，不引入隐藏特权。
  - 政治或道德表达可以降温以降低误读，但不得牺牲平权、规则可改、发起者无特权等核心原则。
- 异常降级:
  - 如果某项安全措施会破坏完全公平性，必须改为更透明、更小权限、更可审计的方案。
  - 如果法律或平台规则风险与实验自由冲突，优先保护项目不被下架、不卷入法律争议、不造成实际伤害。

### Project-Specific-Docs-ConsolidateBeforeCreating

- 状态: 生效
- 适用范围: README、CHARTER、CONTRIBUTING、docs/、治理说明、风险说明和后续公共文档。
- 核心原则: 不为每个新想法新增 Markdown 文档；优先把新规则合并进现有权威文档。只有当内容需要独立生命周期、会被反复引用、篇幅过长或无法自然归入 README/CHARTER/CONTRIBUTING/docs 现有文件时，才新建文档。
- 输入参数:
  - 新增内容的用途。
  - 现有文档结构。
  - 该内容是否会被自动化、贡献者或治理流程反复引用。
- 预期输出:
  - 核心规则优先进入 `CHARTER.md` 与 `CHARTER.en.md`。
  - 项目定位与简短入口优先进入 `README.md` 与 `README.en.md`。
  - 贡献流程优先进入 `CONTRIBUTING.md` 与 `CONTRIBUTING.zh-CN.md`。
  - 较长的专题边界才进入 `docs/`。
- 异常降级:
  - 如果新文档造成维护负担或与现有文档重复，应合并回权威文档并删除或标记过时。
  - 如果文档与脚本能力不一致，必须在文档中明确当前能力边界。

### Project-Specific-Influence-PublicSignalsNotRanking

- 状态: 生效
- 适用范围: Influence Sentinel、Skill 影响力统计、欢迎度展示、生命周期辅助判断、复活机制辅助信息和后续 Agent 维护建议。
- 核心原则: 影响力数据是公开实验信号，不是商店排名、质量裁判或永久身份权力；统计只能来自 Git 历史、仓库文件、Issue/PR/评论/Reactions 等可审计对象，不得引入不可见权重、外部数据库或隐藏人工打分。
- 输入参数:
  - `community/*.md` 的 frontmatter 与 Git 历史。
  - GitHub Issue、PR、评论、Reaction 中公开可读取的互动信号。
  - 复活投票中已经公开记录的支持/反对反应。
- 预期输出:
  - `community/index.json` 作为机器可读公开索引。
  - `docs/impact.md` 作为人类可读实验面板。
  - 输出中必须写明评分公式、生成时间和“非最终价值裁判”的说明。
- 异常降级:
  - 如果 GitHub API 不可读取，应降级为只使用本地 Git 与文件信号，并在日志中说明，而不是阻断仓库运行。
  - 如果某项信号被发现容易被滥用，应优先调整公开公式或降低权重，不得改成隐藏算法。
  - 影响力分数不得直接导致封禁、剥夺复活权或改变作者平等身份，只能作为观察、排序和维护建议。

## 专属 Skill 生成规则

只有同时满足以下条件，才追加新的 Project-Specific Skill：

1. 现有通用 Skill 与本文件已有 Skill 不能精确覆盖当前逻辑。
2. 该逻辑具有高复杂度，或预计在本项目中复用 2 次以上。

新增 Skill 命名必须使用 `Project-Specific-[模块名]-[功能名]`。新增 Skill 必须写明状态、适用范围、输入参数、预期输出和异常降级。

## 审计与熔断规则

- 每完成一个大的功能模块后，审查本文件是否存在过时、重叠或异常 Skill。
- 如果某条 Skill 导致逻辑错误，立即在当前会话降级其优先级，并将其状态标记为 `[状态: 异常/待修复]`，附带原因；在人工明确修正前暂停调用。
